## Introduction

TMT uses Firebase Authentication as its primary authentication mechanism. The system supports three distinct user types, each with specific permissions and access levels:

- **Clients** (`u_clients`) - Event organizers and ticket buyers
- **Collaborators** (`u_collaborators`) - Users associated with specific clients
- **Staff** (`u_staff`) - Internal TMT platform administrators

## Firebase Auth Integration

All user creation flows leverage Firebase Authentication's `createUser` method to generate unique UIDs. These UIDs serve as document IDs in Firestore collections, creating a direct link between Firebase Auth and user data.

<CodeGroup>
```javascript Authentication Flow
// Firebase creates the user
const userRecord = await auth.createUser({
  email: email,
  password: password
});

// UID is used as Firestore document ID
const uid = userRecord.uid;
await db.collection('u_clients').doc(uid).set(userData);
```
</CodeGroup>

## Authenticating API Requests

### HTTP Endpoint Authentication

For HTTPS callable functions, include the Firebase ID token in the request:

<CodeGroup>
```javascript Client-side
const idToken = await firebase.auth().currentUser.getIdToken();

const response = await fetch('https://your-region-project.cloudfunctions.net/endpoint', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${idToken}`
  },
  body: JSON.stringify({
    data: { /* your data */ }
  })
});
```

```javascript Callable Functions
// For Cloud Functions v2 callable functions
const result = await firebase.functions().httpsCallable('functionName')({
  param1: value1,
  param2: value2
});
```
</CodeGroup>

## Session Management

TMT implements custom session management using Firebase custom claims and Firestore.

### Setting Session IDs

The `setSessionId` callable function manages user sessions:

<CodeGroup>
```javascript Client-side
const setSession = firebase.functions().httpsCallable('setSessionId');

const result = await setSession({
  sessionId: 'unique-session-identifier'
});
```

```javascript Server Implementation
// From validate_user.js:60-73
exports.setSessionId = functions.https.onCall(async (data, context) => {
  if (!context.auth) throw new functions.https.HttpsError("unauthenticated");

  const uid = context.auth.uid;
  const sessionId = data.sessionId;

  // Save to Firestore
  await admin.firestore().collection("users").doc(uid).set(
    { sessionId }, 
    { merge: true }
  );

  // Set as custom claim
  await admin.auth().setCustomUserClaims(uid, { sessionId });

  return { success: true };
});
```
</CodeGroup>

<Note>
  The session ID is stored both in Firestore and as a Firebase custom claim, providing dual-layer session validation.
</Note>

### How It Works

1. Client generates or receives a session ID
2. Session ID is stored in Firestore under the user document
3. Session ID is also set as a custom claim on the Firebase Auth token
4. Custom claims are included in the ID token and can be verified server-side

<Warning>
  Custom claims may take up to an hour to propagate. Force token refresh on the client after setting session IDs:
  
  ```javascript
  await firebase.auth().currentUser.getIdToken(true); // Force refresh
  ```
</Warning>

### CORS Configuration

<Warning>
  The current implementation uses `origin: "*"` which allows all origins. In production, restrict CORS to specific domains:
  
  ```javascript
  const cors = require("cors")({
    origin: ["https://yourdomain.com", "https://app.yourdomain.com"]
  });
  ```
</Warning>

### Authentication Context

For callable functions, authentication context is automatically provided:

<CodeGroup>
```javascript Checking Authentication
exports.myFunction = functions.https.onCall(async (data, context) => {
  // Check if user is authenticated
  if (!context.auth) {
    throw new functions.https.HttpsError(
      'unauthenticated',
      'User must be authenticated'
    );
  }
  
  // Get user UID
  const uid = context.auth.uid;
  
  // Access custom claims
  const sessionId = context.auth.token.sessionId;
  
  // Your function logic
});
```
</CodeGroup>

### Input Validation

<Tip>
  Always validate user input on the server side. Check for required fields and data types:
  
  ```javascript
  if (!data.email || !data.uid_user) {

  }
  ```
