## Descripción

Esta función HTTP crea una nueva orden en el sistema TMT, incluyendo los tickets comprados y las transacciones de pago asociadas. La orden se almacena tanto en Firestore como en PostgreSQL.

<Note>
  Al crear una orden con `status: true`, se activa automáticamente el trigger `order_process` que:
  - Genera las transacciones individuales
  - Actualiza los tickets con información del cliente
  - Crea el registro en u_users
  - Ejecuta el split de pagos según la configuración financiera del evento
</Note>

## Endpoint

```
POST https://your-region-your-project.cloudfunctions.net/order_created
```

## Estructura de Datos en Firestore

La orden creada en Firestore contiene:

```javascript
{
  amount: 150.00,
  date: {
    created: Timestamp,
    updated: ''
  },
  event_id: "evt_abc123",
  event_name: "Concierto Rock 2024",
  office_id: "ofc_xyz789",
  office_name: "Oficina Central",
  status: true,
  status_type: {
    id: "completed",
    name: "Completada"
  },
  tickets: [...],
  transactions: [...],
  client_id: "client_456",
  client_name: "Productora Musical XYZ",
  exchange_rate: 36.50,
  box_office_name: "Taquilla Principal",
  box_office_id: "box_001",
  purchaser_info: {...},
  recipient_info: null,
  is_courtesy: false,
  is_corporate: false,
  is_gift: false
}
```

<Warning>
  Las transacciones se almacenan con timestamps automáticos. El campo `date.created` se establece con `Timestamp.now()` al momento de la creación.
</Warning>

## Proceso Automático Post-Creación

Cuando se crea una orden con `status: true`, el sistema ejecuta automáticamente:

1. **order_process** (Firestore Trigger)
   - Valida el monto total vs suma de transacciones
   - Actualiza tickets con información del cliente
   - Genera transacciones individuales
   - Crea/actualiza registro en u_users
   - Ejecuta el split de pagos según configuración financiera

2. **Split de Pagos** (solo si `point_sale_tmt: false`)
   - Calcula costos fijos y variables
   - Crea registros en orders_payout
   - Distribuye entre TMT y Cliente según porcentajes configurados

3. **Conciliación** (si `point_sale_tmt: true`)
   - Envía transacción a proceso de conciliación
   - El split se ejecuta después de conciliar

## Casos de Uso

- Venta de tickets en taquilla física
- Venta en línea a través del portal
- Ventas corporativas
- Emisión de cortesías
- Ventas con múltiples métodos de pago
- Ventas en múltiples monedas

## Consideraciones

- El `status_type` siempre se establece como "completed" al crear la orden
- Se requiere al menos una transacción válida
- El `exchange_rate` debe ser mayor a 0
- Los IDs de tickets deben seguir el formato "event_id-ticket_id"
- La función guarda tanto en Firestore como en PostgreSQL para redundancia
