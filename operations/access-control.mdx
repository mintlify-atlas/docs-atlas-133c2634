## Overview

The access control system manages credential-based entry and exit tracking for events. It supports both real-time and offline (cold) access control operations, enabling staff to monitor attendee flow and validate credentials at venue checkpoints.

## Access Control Workflow

<Steps>
  <Step title="Checkpoint Assignment">
    Collaborators are assigned to event checkpoints with specific access permissions
  </Step>
  
  <Step title="Credential Validation">
    When a credential is scanned, the system validates it against the event's credential database
  </Step>
  
  <Step title="Entry/Exit Tracking">
    The system tracks entry and exit status with real-time updates
  </Step>
  
  <Step title="Cold Synchronization">
    Offline access events are synchronized in batches when connectivity is restored
  </Step>
</Steps>

## Functions

### events_list_checkpoints

Retrieves all active checkpoints assigned to a collaborator.

**Endpoint:** `POST /events_list_checkpoints`

**Request Body:**

```json
{
  "data": {
    "userd_id": "collaborator_id"
  }
}
```

**Implementation:**

```javascript
await db.collection("u_collaborators").doc(userd_id)
  .collection("checkpoints")
  .where("status", "in", ['Activo', 'En Progreso'])
  .get()
  .then(async (snapshot) => {
    if (snapshot._size > 0) {
      let responses = [];
      snapshot.forEach((doc) => {
        let data = doc.data();
        let resp = {
          "event_id": doc.id,
          "date_end": data.date_end,
          "date_start": data.date_start,
          "type": data.type,
          "status": data.status,
          "event": data.event.media,
          "name": data.name
        };
        responses.push(resp);
      });
    }
  });
```

**Response:**

```json
{
  "message": "Eventos encontrados",
  "status": 200,
  "data": {
    "valido": true,
```

## Access State Machine

<Steps>
  <Step title="Never Entered">
    `access_status: false`, `access_entry: false`
    
    Initial state when credential is created
  </Step>
  
  <Step title="First Entry">
    `access_status: true`, `access_entry: true`
    
    Holder enters venue for the first time
  </Step>
  
  <Step title="Exited">
    `access_status: true`, `access_entry: false`
    
    Holder has exited the venue
  </Step>
  
  <Step title="Re-Entry">
    `access_status: true`, `access_entry: true`
    
    Holder re-enters after exiting
  </Step>
</Steps>

## Best Practices

<CardGroup cols={2}>
  <Card title="Real-Time Operations" icon="signal">

    - Use clear visual indicators for entry states
    - Handle network failures gracefully
  </Card>
  
  <Card title="Offline Operations" icon="wifi-slash">
    - Store access events locally when offline
    - Use cold synchronization for batch updates
    - Deduplicate events before syncing
    - Preserve chronological order in ledger
  </Card>

<Warning>
  The cold synchronization function processes multiple events per ticket. It deduplicates by ticket ID and only processes the final state from the batch. Ensure events are ordered correctly.
</Warning>

## Checkpoint Management

Checkpoints are assigned to collaborators and filtered by status:

- **Activo**: Checkpoint is active and ready for use
- **En Progreso**: Checkpoint is currently in use
- **Inactivo**: Checkpoint is deactivated (not returned in queries)

<Note>
  Only checkpoints with status `Activo` or `En Progreso` are returned when listing available checkpoints for a collaborator.
</Note>

## Ledger Actions for Access Control

| Action | Description | Trigger |
|--------|-------------|----------|
| `accessed` | Entry recorded | Credential scanned at entry |
| `came-out` | Exit recorded | Credential scanned at exit |

## Ticket vs Credential

The system supports both ticket-based and credential-based access:

- **Credentials**: Used for staff, VIP, and special access (stored in `events/{event_id}/credentials`)
- **Tickets**: Used for regular attendees (stored in `events/{event_id}/tickets` with ledger tracking)

Both use the same access state model (`access_status` and `access_entry`).

## Cold Synchronization Details

<Steps>
  <Step title="Ticket Validation">
    Verify tickets exist and are not sold (`status = false`)
  </Step>
  
  <Step title="Deduplication">
    Process each unique ticket_id only once, considering all events for that ticket
  </Step>
  
  <Step title="Ledger Merge">
    Combine existing ledger with new access events
  </Step>
  
  <Step title="Final State">
    The last event type (in/out) determines the final `access_entry` value
  </Step>
  
  <Step title="Dual Update">
    Update both Firestore and PostgreSQL in sequence
  </Step>
</Steps>

<Tip>
  When implementing offline access control, collect all entry/exit events locally and send them in a single batch to the cold synchronization endpoint when connectivity is restored.
</Tip>

## Error Scenarios

- **Invalid credential**: Credential document not found
- **Already inside**: Attempting entry when `access_entry` is already `true`
- **Not inside**: Attempting exit when `access_entry` is `false`
- **Inactive credential**: Credential exists but `status` is `false`
- **No checkpoints**: Collaborator has no active checkpoints assigned
