## Overview

The offline mode system allows ticket offices to operate without continuous internet connectivity by assigning tickets to specific offices for offline sales. The system handles ticket assignment, status tracking, and synchronization back to the main database.

## Offline Workflow

<Steps>
  <Step title="Ticket Assignment">
    Assign available tickets to an office for offline operation
  </Step>
  
  <Step title="Offline Sales">
    Tickets are marked as offline and can be sold without real-time connectivity
  </Step>
  
  <Step title="Synchronization">
    When connectivity returns, offline sales are synchronized with the main system
  </Step>
  
  <Step title="Unassignment">
    Unused tickets can be returned to the online pool
  </Step>
</Steps>

## Data Structures

### Ticket Ledger

Each ticket maintains a ledger tracking all state changes:

```javascript
{
  "date": Timestamp,
  "action": "offline" | "unassign" | "accessed" | "came-out",
  "metadata": "{}"
}
```

### Offline Ticket Record

```javascript
{

  "date_created": Timestamp,
  "event_id": "string",
  "office_id": "string"
}
```

**SQL Query:**

```sql
SELECT 
  id, color, event_description, t.event_id, seat_id, 
  CASE 
    WHEN status_sold THEN false 
    WHEN status THEN true

  t.ticket_id, zone, event_name, seat_row, ledger,
```

## Synchronization Process

<Steps>
  <Step title="Dual Database Update">
    Updates are applied to both Firestore and PostgreSQL databases simultaneously using batch operations
  </Step>
  
  <Step title="Ledger Tracking">
    All ticket state changes are recorded in the ledger array with timestamps and action types
  </Step>
  
  <Step title="Status Flags">
    The `status_offline` flag indicates whether a ticket is currently assigned to an office for offline operation
  </Step>
  
  <Step title="Atomic Operations">
    Batch commits ensure all updates succeed or fail together, maintaining data consistency
  </Step>
</Steps>

## Best Practices

<CardGroup cols={2}>
  <Card title="Assignment Strategy" icon="ticket">

## Ticket States

| State | Description | `status` | `status_offline` |
|-------|-------------|----------|------------------|
| Available Online | Ticket available for online purchase | `true` | `false` |
| Assigned Offline | Ticket assigned to office | `true` | `true` |
| Sold Offline | Ticket sold through office | `false` | `true` |
| Blocked | Ticket temporarily blocked | varies | varies |

## Database Operations

The offline mode system uses dual-database architecture:

### Firestore Operations

```javascript
var batch = db.batch();
batch.update(
  db.collection('events').doc(event_id).collection('tickets').doc(ticket_id),
  { "status_offline": true, "ledger": responses }
);
await batch.commit();
```

### PostgreSQL Operations

```javascript
await dbpostgres.sqltmt(
  "update", 
  "tickets", 
  "status_offline=tmp.status_offline, ledger=tmp.ledger",
  "ticket_id = tmp.ticket"
);
```

<Tip>
  Always update both databases in sequence. Use batch operations in Firestore and multi-row updates in PostgreSQL for optimal performance.
</Tip>

## Error Handling

- **Unavailable tickets**: Returns list of tickets that cannot be assigned (already sold or offline)
- **Blocked tickets**: Checks separate `tickets_blocked` table to prevent assignment
- **Invalid tickets**: Non-existent ticket IDs are caught during validation
- **Sync failures**: Monitor callable function responses for order creation

## Ledger Actions

| Action | Description | Context |
|--------|-------------|----------|
| `offline` | Ticket assigned to office | Assignment |
| `unassign` | Ticket returned to online | Unassignment |
| `accessed` | Ticket scanned for entry | Access control |
| `came-out` | Ticket holder exited venue | Access control |
